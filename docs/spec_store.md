# 매장(Store) 관리 시스템 기획 명세서

본 문서는 관리자 대시보드 내 **매장(Store) 관리 카테고리**의 프론트엔드 및 백엔드 개발 시 필요한 세부 필수 항목과 페이지 프로세스를 정의한 문서입니다. 매장은 영업 상태와 배달 설정의 실시간 연동이 매우 중요한 도메인입니다.

## 2. 페이지 프로세스 (Page Process)

1. **매장 목록 조회** (`StoreList`) — 현재 운영중(`active`)인 매장을 우선 노출한다. 지역, 계약 상태, 매장명, 점주명 조합 필터를 제공한다.
2. **매장 등록/수정** (`StoreForm`) — 탭 형태의 복합 입력 폼(기본정보, 영업/배달설정, 연동, 노출)으로 구성한다.
3. **매장 상세** (`StoreDetail`) — 읽기 전용 정보 표시. 하위 편집 페이지 진입점 제공. 영업상태는 영업시간/휴무일 룰로 자동 계산되므로 수동 상태 토글 없음.
4. **영업정보 수정** (`OperatingInfoEdit`) — 5개 독립 카드 섹션으로 구성: ①운영 상태(노출+임시휴업 토글) ②영업시간(평일/주말/공휴일/요일별) ③배달 설정(최소주문금액 직접 설정) ④포장 설정(최소주문금액 직접 설정) ⑤예약 설정. 임시휴업은 단일 토글+사유 선택만 제공하며, 다음날 자동 해제된다.
5. **휴무일 수정** (`ClosedDayEdit`) — 정기휴무(weekly/monthly_nth/monthly_date)와 비정기휴무(날짜+사유) 관리.
6. **편의시설 수정** (`AmenitiesEdit`) — 주차(주차 안내문), 매장식사(좌석수), 와이파이, 드라이브스루 지원 여부 설정.
7. **결제수단 수정** (`PaymentMethodsEdit`) — 현금/카드/간편결제 허용 여부 관리.
8. **연동코드 수정** (`IntegrationCodesEdit`) — POS/PG 연동 코드 관리. 일괄 업로드(PG/POS Bulk Upload Modal) 지원.
9. **노출 설정** — 웹/모바일 공통 단일 매장 노출 on/off 관리. 영업정보 수정 페이지의 "운영 상태" 카드에 통합.
10. **가맹점주** — 가맹점주는 초대 승인 완료 시 자동으로 매장에 연결된다. 수동 직원 연결 기능 삭제됨. 매장당 1명의 점주만 연결 가능(1:1). 기존 점주가 있는 매장에 새 점주를 초대하려면 기존 연결을 먼저 해제해야 한다.

| 기능 / 필드명 | 입력/노출 형태 | 필수 여부 | 글자수 / 제약조건 | 비고 (UI/UX) |

### 3.2. 백엔드 (Backend) 개발 요건

#### Store 테이블

| 데이터베이스 필드 | 데이터 타입 | 필수 여부 | 글자수 / 제약조건 | 비고 (API 설계) |
| :--- | :--- | :---: | :--- | :--- |
| **매장 고유번호 (id)** | 고유식별자 | 필수 | 36자 | 시스템에서 자동 생성되는 매장 식별 번호이다. |
| **매장명 (name)** | 문자열 | 필수 | 최대 50자 | 매장 이름이다. |
| **매장 상태 (status)** | 선택값 | 필수 | 4종 | 운영중(active) / 비활성(inactive) / 대기(pending) / 계약해지(terminated) 중 하나이다. |

| **주소 (address)** | 묶음 데이터 | 필수 | - | 우편번호, 주소, 상세주소, 위도, 경도를 포함한다. |
| **점주 정보 (ownerInfo)** | 묶음 데이터 | 필수 | - | 점주 이름, 전화번호, 이메일을 포함한다. |
| **사업자 정보 (businessInfo)** | 묶음 데이터 | 필수 | - | 사업자번호, 상호명, 대표자명을 포함한다. 매장 등록 시 직접 입력한다. |
| **계약 정보 (contractInfo)** | 묶음 데이터 | 필수 | - | 브랜드 본사가 해당 매장(가맹점)을 관리하기 위한 계약 데이터. 계약 유형(가맹/직영/라이선스), 계약일, 만료일, 계약 상태, 비고를 포함한다. ~~로열티 비율, 보증금 삭제됨.~~ |
| **정산 계좌 (bankAccountInfo)** | 묶음 데이터 | 필수 | - | 은행명, 계좌번호, 예금주명을 포함한다. |
| **영업 정보 (operatingInfo)** | 묶음 데이터 | 필수 | - | 평일/주말/공휴일/요일별 영업시간, 정기/비정기 휴무, 임시휴업(단일 토글+사유, 다음날 자동 해제), 배달/포장 가능 여부, 매장 노출 여부를 포함한다. 앱 운영 상태는 영업시간/휴무일 룰로 자동 계산된다. ※ 배달비는 상권관리(DeliveryZone)에서 관리하므로 여기에 포함하지 않는다. |
| ~~노출 설정 (visibilitySettings)~~ | - | - | - | **삭제됨**: 매장 노출은 영업정보(OperatingInfo.isVisible) 단일 필드로 관리한다. 채널별 노출, 뱃지, 추천 매장 기능 폐기. |
| **편의시설 (amenities)** | 묶음 데이터 | 선택 | - | 주차 가능 여부+안내문, 매장식사 가능 여부+좌석수, 와이파이 가능 여부를 포함한다. |
| **결제수단 (paymentMethods)** | 묶음 데이터 | 선택 | - | 카드/현금/포인트 허용 여부, 간편결제(카카오페이/네이버페이/토스페이 등) 개별 허용 여부를 포함한다. |
| **연동 코드 (integrationCodes)** | 묶음 데이터 | 선택 | - | POS(포스기) 연동 정보, PG(결제대행사) 연동 정보. **슈퍼어드민에서 상속**된다. |
| ~~배달비 (deliveryFee)~~ | - | - | - | **삭제됨**: 배달비는 상권관리에서 단일 관리한다. 메인상권 배달비 = 기본, 추가상권 = 메인 배달비 + 추가 배달비. |

#### 계약 정보 (contractInfo) 상세 — 본사↔매장 간 가맹 계약

| 필드명 | 타입 | 필수 | 설명 |
| :--- | :--- | :---: | :--- |
| **계약 유형 (contractType)** | 선택값 | 선택 | 가맹(franchise) / 직영(direct) / 라이선스(license). 기본값: 가맹 |
| **계약일 (contractDate)** | 날짜 | 필수 | 가맹 계약 시작일 |
| **만료일 (expirationDate)** | 날짜 | 필수 | 가맹 계약 만료일 |
| **계약 상태 (contractStatus)** | 선택값 | 필수 | 정상(active) / 만료(expired) / 갱신대기(pending_renewal) |
| **비고 (notes)** | 문자열 | 선택 | 계약 관련 메모, 특이사항, 별도 약정 등 |

> **참고**: 이 계약 정보는 플랫폼(슈퍼어드민)과 브랜드 간 계약이 아니라, **브랜드 본사가 소속 매장(가맹점)을 관리하기 위한 계약 데이터**이다. 플랫폼↔브랜드 간 계약은 슈퍼어드민에서 별도 관리된다.

**[운영 규칙 및 제약사항]**
- **영업상태 자동 계산** — 영업시간/휴무일 룰에 따라 자동 결정된다(open/break_time/closed/temporarily_closed). 관리자 수동 전환 기능 없음. 임시휴업은 토글 ON 시 당일 적용, 다음날 00:00 자동 해제(배치).
- **예약 시간 검증** — 예약 가능 시간(N분 후부터)이 매장 마감 시간을 넘지 않도록 서버에서 자동 검증한다.
- **점주 연결 규칙 (1:1)** — 각 매장은 정확히 1명의 점주 계정과만 연결된다. 가맹점주 초대 승인 시 자동으로 연결된다. 해당 매장에 이미 점주가 등록되어 있으면 초대 시 에러 발생(기존 계정 삭제 후 초대 안내).
- **점주 연결 해제** — 매장 상세 > 가맹점주 탭에서 기존 점주 연결 해제 가능. 새 점주 초대 전 반드시 해제 필요.
- **영업정보 저장 방식** — 영업정보는 전체를 한 번에 저장하는 방식이다. 일부만 수정해도 전체 데이터를 함께 보낸다.
- **배달/포장 최소 1개 필수** — 배달과 포장을 동시에 끄는 것은 불가하다. 서버와 화면 양쪽에서 검증한다.
- **주문 가능 시간** — 별도 설정 없이 "영업 시작 시각 ~ (마감 시각 - 라스트오더 분)"으로 자동 계산된다. (예: 영업 09:00~22:00, 라스트오더 30분 → 주문 가능: 09:00~21:30)

**[⚠️ 트래픽/성능 검토]**
- **앱 운영 상태 조회** — 영업시간/휴무일 룰 기반으로 자동 계산. Redis에 매장별 운영 상태 캐싱, 영업시간 경계 시점에 갱신한다. 임시휴업 자동 해제 배치(매일 00:00).
- **매장 목록 API** — 지역+상태 조합 조회가 빈번하므로 `(region, status)` 복합 인덱스를 생성한다.
- **POS 연동 테스트** — 외부 API 호출이므로 타임아웃(5초)과 Circuit Breaker를 적용한다.

## 3.3. 권한 기반 접근 제어

권한별 규칙은 고정 역할 대신 개별 권한 키(permission key)에 기반한다. 각 사용자는 권한 키를 부여받으면 해당 기능에 접근 및 수정 가능하다.

| 권한 키 | 조회 (Read) | 수정 (Write) | 비고 |
| :--- | :---: | :---: | :--- |
| `store:read` | ✅ | ❌ | 매장 정보 조회만 가능. 수정 불가. |
| `store:write` | ✅ | ✅ | 기본정보, 영업정보, 편의시설, 결제수단 수정 가능. |
| `store:admin` | ✅ | ✅ | 계약정보, 연동코드, 노출설정 등 전체 관리 가능 (최상위 권한). |
| 권한 없음 | ❌ | ❌ | 접근 차단 (403 반환). |

**룰 적용 원칙:**
- 권한은 역할(role)에 종속되지 않으며, 계정별 개별 설정된다.
- `store:write`는 `store:read` 권한을 포함한다.
- `store:admin`은 `store:write` 권한을 포함한다.
- 매장 범위(담당 매장만 vs. 전체 매장)는 `scope: own|all` 필드로 별도 제어된다.

## 5. 개발자용 정책 설명

### 5.1. 매장 노출 정책

```
■ 노출 설정: OperatingInfo.isVisible 단일 필드 (웹/모바일 공통)
  · isVisible=false → 고객앱 검색/목록에서 완전 비노출
  · 채널별 노출, 뱃지, 추천 매장 기능은 폐기됨
  · VisibilitySettings 인터페이스 삭제 — Store.visibilitySettings 필드 없음

■ 임시 휴업 (단일 토글 — 다음날 자동 해제)
  · isTemporarilyClosed=true → 고객앱에서 "휴업중" 라벨 표시, 주문 접수 차단
  · temporaryCloseReason 필수 입력 (재료 소진/기기점검/개인사정/기타)
  · 날짜 지정 없음 — 당일만 적용, 다음날 00:00 배치로 자동 해제 (isTemporarilyClosed=false)
```

### 5.2. 배달비 정책 (Single Source of Truth: 상권관리)

```
배달비는 상권관리(DeliveryZone)에서 단일 관리한다.
매장 영업정보(OperatingInfo)에서는 배달비를 설정하지 않는다.

■ 배달비 계산 규칙
  · 메인상권: DeliveryZone.deliveryFee = 기본 배달비
  · 추가상권: 고객 청구 배달비 = 메인상권 배달비 + 추가상권 deliveryFee
  · 무료배달: 매장의 DeliverySettings.minOrderAmount 이상 주문 시 배달비 면제

■ 데이터 위치
  · 상권별 배달비: DeliveryZone.deliveryFee (상권관리에서 설정)
  · 최소주문금액: Store.DeliverySettings.minOrderAmount (매장관리에서 설정)
  · 포장 최소주문금액: Store.PickupSettings.minOrderAmount (매장관리에서 설정)
  · 브랜드 기본값: InitialSettings.deliveryFee (super-admin, 읽기 전용)
  · 주문 청구액: Order.deliveryFee (주문 시점 스냅샷, 정산 집계용)
  · ※ Store.OperatingInfo에는 배달비 필드 없음 (삭제됨)
  · ※ DeliveryZone.minOrderAmount 삭제됨 — 최소주문금액은 매장에서 관리
```

# 배달권역(Delivery Zone) 관리 시스템 기획 명세서

본 문서는 관리자 대시보드 내 **배달권역 관리** 페이지의 프론트엔드 및 백엔드 개발 시 필요한 세부 필수 항목과 페이지 프로세스를 정의한 문서다.

## 1. 라우트 구조

| 경로 | 페이지 | 권한 |
| :--- | :--- | :--- |
| `/delivery-zones` | 배달권역 목록 (DeliveryZoneList) | staff:read |
| `/delivery-zones/new` | 배달권역 등록 (DeliveryZoneEditor) | staff:write |
| `/delivery-zones/:id/edit` | 배달권역 수정 (DeliveryZoneEditor) | staff:write |

---

## 2. 페이지 프로세스

1. **상권 목록 조회** — 매장별 배달 가능 상권을 목록으로 조회한다. 좌측에 카드 목록, 우측에 지도를 표시한다. 매장 필터로 필터링 가능.
2. **상권 등록/수정** — 3패널 레이아웃(좌측 폼 + 중앙 지도 + 우측 상권 리스트)으로 구성한다.
   - **반경 모드** — 지도에서 중심점 클릭 → 반경 클릭으로 원형 영역을 지정한다.
   - **폴리곤 모드** — 지도에서 꼭짓점을 클릭하여 다각형 영역을 지정한다. 최소 3개 점 필요. 첫 점 근처 클릭으로 완성.
3. **소상권 자동 생성 (거리별 배달비)** — **반경 모드 전용**. 메인 상권(반경) 설정 후 "소상권 사용" 체크 시, 거리 간격(N미터)을 입력하여 동심원 소상권을 자동 생성한다.
   - 예: 메인 3km, 간격 500m → 0~500m / 500m~1km / ... / 2.5km~3km (6개 구간)
   - 각 구간별 추가 배달비를 모달에서 설정한다. (고객 청구 배달비 = 메인 배달비 + 해당 구간 추가 배달비)
   - 저장 후 우측 패널에 소상권 목록이 표시되며, 개별 수정이 가능하다.
4. **소상권 수동 생성** — **반경/폴리곤 모두 지원**. 상권 구분을 "소상권"으로 선택 후 직접 반경 또는 폴리곤을 그려서 소상권을 수동으로 생성한다. 수동 생성 시 소상권의 크기(반경)를 기본상권(메인상권)보다 크게 그릴 수 있다.

| **소상권 리스트** | 우측 패널 카드 | - | - | 저장된 소상권 목록 표시. 거리 구간(0m~500m) + 추가배달비 + 총 배달비 + [수정] 버튼. |

**[운영 규칙 및 제약사항]**
- **폴리곤 좌표** — 꼭짓점 최소 3개, 최대 제한 없음.
- **상권 중복 허용** — 서로 다른 매장의 상권이 겹칠 수 있다. 같은 브랜드 내 여러 매장이 배달 권역을 공유하는 운영이 존재하기 때문이다. 중복 체크 API는 참고용(경고)이며, 등록을 차단하지 않는다.
- **소상권 소속** — 소상권은 반드시 메인 상권에 연결되어야 한다 (parentZoneId 필수).
- **소상권 일괄 생성** — 소상권 일괄 생성 시 기존 소상권을 모두 삭제 후 새로 생성한다. 수정이 아닌 전체 교체 방식.
- **상권 유형** — 'radius'(반경)와 'polygon'(폴리곤) 2가지만 지원. 기획서의 'circle' 타입은 radius로 통합됨.
- **배달비 계산** — 메인 상권 배달비 + 소상권 추가 배달비 = 고객 청구 배달비. 소상권이 없으면 메인 배달비만 적용.
- **최소 주문금액** — 매장관리(영업정보)에서 배달/포장 각각 설정. 상권에서는 관리하지 않는다.

**[⚠️ 트래픽/성능 검토]**
- **주문 시 권역 판별** — 고객 주소 좌표가 어느 권역에 포함되는지 판별하는 로직이 주문마다 호출된다. PostGIS의 ST_Contains 또는 Ray Casting 알고리즘을 사용하되, 매장별 권역 데이터를 Redis에 캐싱하여 DB 부하를 줄인다.
- **지도 렌더링** — 다각형 좌표가 많은(50점) 경우 프론트엔드 Canvas 성능에 주의한다. 간소화(Douglas-Peucker) 적용을 권장한다.

## 5. 개발자용 정책 설명

### 5.1. 권역 중첩 정책

```
■ 동일 매장 내 중첩
  한 매장에 메인 상권이 여러 개 존재할 수 있다.
  중첩 시: 배달비가 가장 낮은 권역을 우선 적용한다.
  고객 화면: 최종 적용 배달비만 표시 (중첩 사실 비노출).

■ 서로 다른 매장 간 중첩
  같은 브랜드 내 여러 매장의 상권이 겹칠 수 있다.
  예: A매장 3km 반경과 B매장 3km 반경이 겹치는 지역 →
     고객 주소가 양쪽 모두 포함되면, 두 매장 모두 주문 가능 매장으로 노출한다.
  브랜드별로 권역 공유 여부가 다르므로 시스템에서 강제 차단하지 않는다.

■ 중복 체크 API
  POST /api/delivery-zones/check-overlap
  → 참고용 경고만 반환, 등록을 차단하지 않음
  → "A매장 상권과 N㎡ 중첩됩니다" 형태의 안내 메시지
```

Point-in-Polygon 알고리즘 사용
고객 주소 → 위경도 변환(Geocoding API) → 폴리곤 포함 여부 판정
Geocoding 실패 시: "주소를 확인할 수 없습니다" 안내
