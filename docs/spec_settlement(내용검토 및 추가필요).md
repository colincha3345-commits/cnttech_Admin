# 정산(Settlement) 관리 시스템 기획 명세서

본 문서는 관리자 대시보드 내 **정산관리 카테고리**(정산 내역, 통계/조회)의 프론트엔드 및 백엔드 개발 시 필요한 세부 필수 항목과 페이지 프로세스를 정의한 문서다.

---

## 0. 라우트 구조

| 경로 | 페이지 | 권한 |
| :--- | :--- | :--- |
| `/settlement` | 정산 목록 (SettlementList) | settlement:read |
| `/settlement/stats` | 정산 통계 (SettlementStats) | settlement:read |
| `/settlement/:id` | 정산 상세 (SettlementDetail) | settlement:read |


## 1. 페이지 프로세스 (Page Process)

### 1.1 정산 내역 (`/settlement`)

1. **정산 목록 조회** — 정산 기간, 매장, 상태(정산전/정산완료) 필터를 적용하여 목록을 조회한다. 매장별 총매출, 할인, 수수료, 실정산액을 요약 컬럼으로 노출한다.
2. **정산 상세** — 행 클릭 시 상세 페이지로 이동한다. 정산 구성 요소(매출, 배달비, 할인, 본사지원금, 포인트, 쿠폰, 교환권, 플랫폼수수료)를 표 형태로 분해하여 보여준다.
3. **주문별 상세** — 정산에 포함된 개별 주문 내역을 하단 테이블에 나열한다. 주문번호, 일시, 메뉴, 금액, 결제수단, 실정산액을 표기한다.
4. **상태 확인** — 정산전(pending) 상태의 건은 정산 배치 실행 후 자동으로 정산완료(completed)로 전환된다.

### 1.2 통계/조회 (`/settlement/stats`)

1. **기간별 통계** — 월별/주별/일별 총매출, 총정산, 수수료 추이를 차트와 테이블로 제공한다.
2. **매장별 비교** — 매장별 정산 금액을 비교하는 랭킹 테이블을 노출한다.
3. **엑셀 내보내기** — 조회된 통계 데이터를 다운로드한다.

- 정산 상세의 금액 필드는 모두 ReadOnly이며, 천 단위 콤마와 원(₩) 단위를 표기한다.

### 2.2. 백엔드 (Backend) 개발 요건

| 데이터베이스 필드 | 데이터 타입 | 필수 여부 | 글자수 / 제약조건 | 비고 (API 설계) |
| :--- | :--- | :---: | :--- | :--- |
| **id (PK)** | UUID | Y | 36자 | 정산 고유 식별자다. |
| **storeId (FK)** | UUID | Y | - | 정산 대상 매장 참조다. |
| **period** | String | Y | - | "YYYY-MM-DD ~ YYYY-MM-DD" 형식이다. |
| **totalSales** | Integer | Y | 0 이상 | 기간 내 총매출이다. |
| **deliveryFee** | Integer | Y | 0 이상 | 배달비 합계다. |
| **promotionDiscount** | Integer | Y | 0 이상 | 전체 할인 총액이다. (쿠폰 할인 + 포인트 할인 + 프로모션 할인) |
| **hqSupport** | Integer | Y | 0 이상 | 본사 부담 할인금이다. (할인액 × 본사 부담 비율) |
| **pointsUsed** | Integer | Y | 0 이상 | 고객 적립 포인트 결제 합계다. |
| **couponsUsed** | Integer | Y | 0 이상 | 쿠폰 할인 합계다. (본사/가맹 발행 구분) |
| **vouchersUsed** | Integer | Y | 0 이상 | 모바일 상품권·교환권 결제 합계다. |
| **platformFee** | Integer | Y | 0 이상 | 플랫폼 수수료다. (매출 × super-admin FeePolicy.baseRate) |
| **pgFee** | Integer | Y | 0 이상 | PG사 결제 수수료다. (매출 × super-admin FeePolicy.pgFeeRate) |
| **netAmount** | Integer | Y | 통상 0 이상 | 최종 실정산액이다. 정률 수수료 구조상 수수료가 매출을 초과하지 않으므로 통상 음수 불가. 전기 취소/환불 차감이 당기 매출 초과 시 예외적 음수 가능. 시스템 차단 없음, 발생 시 운영자 알림. |
| **status** | 선택값 | 필수 | 2가지 | 'pending'(정산전), 'completed'(정산완료). 정산일 기준으로 자동 전환. |
| **paymentDate** | Date | N | - | 지급 완료일이다. |
| **orderCount** | Integer | Y | 0 이상 | 기간 내 주문 건수다. |

**[API 및 비즈니스 로직 제약사항]**
- **정산 대상 필터** — 정산 배치는 주문 status='completed'인 주문만 집계한다. payment_pending(결제 대기), expired(결제 만료), cancelled(취소) 상태 주문은 정산에서 제외된다.
- **정산 산출 배치** — 정산 기간(반월/월단위) 종료 시 자동으로 주문 데이터를 집계하여 Settlement 레코드를 생성한다. 계산 공식: `netAmount = totalSales + deliveryFee + hqSupport - promotionDiscount - platformFee - pgFee`. 산출 결과가 0 미만이 되지 않도록 수수료율과 할인 정책을 사전 검증해야 한다.
- **정산액 구성요소 설명** —
  - `totalSales` (매출): 정산 기간 내 완료된 주문의 상품 금액 합계
  - `deliveryFee` (배달비): 배달 주문에서 발생한 배달비 합계
  - `promotionDiscount` (할인 총액): 쿠폰 할인(`couponAmount`) + 포인트 할인(`pointUsed`) + 프로모션 할인(`discountAmount`)의 합계. 이 중 본사/가맹 부담 비율에 따라 분리 정산된다.
  - `hqSupport` (본사 지원금): 할인액 중 본사가 부담하는 금액. 가맹점 입장에서는 수입으로 합산된다. (예: 1만원 할인, 본사 70% 부담 → 7,000원 보전)
  - `pointsUsed` (포인트 사용): 고객이 적립 포인트로 결제한 금액. 마케팅관리 > 포인트 설정에서 정한 본사/가맹 부담 비율(`headquartersRatio`/`franchiseRatio`)에 따라 정산 시 분리 처리. (예: 본사 70% / 가맹 30% → 1,000P 사용 시 본사 700원 부담, 가맹 300원 부담)
  - `couponsUsed` (쿠폰 사용): 쿠폰으로 할인된 금액. 쿠폰별 `headquartersRatio`/`franchiseRatio` 설정에 따라 본사/가맹 부담 분리
  - `vouchersUsed` (E쿠폰 — 교환권/금액권 사용): 모바일 교환권(`exchange`)·금액권(`voucher`) 결제 금액. **정산주체는 외부 쿠폰사**(`ECouponUsage.couponCompany`)이며, 플랫폼/가맹점 정산에서 직접 차감하지 않는다. 외부 쿠폰사와 별도 정산 계약에 따라 처리한다.
  - `platformFee` (플랫폼 수수료): 매출액 × 플랫폼 수수료율(super-admin `FeePolicy.baseRate`)로 산출. 플랫폼 운영 수익
  - `pgFee` (PG사 결제 수수료): 매출액 × PG 수수료율(super-admin `FeePolicy.pgFeeRate`)로 산출. 결제대행사(PG사) 비용. 수수료율은 super-admin에서 브랜드별로 설정
- **음수 정산** — 플랫폼 수수료와 PG 수수료는 매출액 기반 정률(%)이므로, 수수료가 매출을 초과하는 구조적 음수는 발생하지 않는다. 다만 이전 정산 확정 후 발생한 취소/환불 차감이 당기 매출보다 큰 예외 케이스에서 음수가 발생할 수 있으므로, 시스템에서 사전 차단하지 않는다. 음수 발생 시 운영자 확인용 알림(alert)을 남기고, 정산 상세 화면에서 빨간색으로 강조 표시한다.
- **상태 전이** — pending(정산전) → completed(정산완료). 정산 배치 실행 시 자동 전환된다.
- **미완료 주문 자동 완료** — 영업일 기준 자정(00:00)이 지나면, 전일의 미완료 주문(pending/confirmed/preparing/ready)을 자동으로 completed(주문완료)로 전환한다. cancelled/payment_pending/expired 주문은 대상이 아니다.
- **정산 배치 스케줄** — 반월 단위(1~15일/16~말일) 종료 익일 새벽 02:00 자동 실행. 자정(00:00)에 미완료 주문 자동 완료 처리 후 실행된다.
- **정산 후 취소** — 주문은 정산 여부와 관계없이 언제든 취소 가능하다. 시스템에서 기한 차단 없음.
- **마감 후 취소 시 마이너스 주문내역** — 정산 완료 후 취소 발생 시, 원본 정산을 소급 수정하지 않는다. 대신 동일 주문번호로 마이너스 주문내역(금액 음수, 유형 'refund')을 생성하여 다음 정산에서 차감한다. (상세: 4.3절 참고)

**[⚠️ 트래픽/성능 검토]**
- **정산 배치** — 반월/월 단위로 대량 주문 데이터를 집계한다. 마감 후 취소/환불은 다음 정산에 반영한다. (상세: 4.3절 참고)
- **집계 쿼리** — 기간별 SUM 쿼리가 무거우므로 사전 집계 테이블(materialized view) 활용을 권장한다.
- **통계 차트** — 월별/주별 통계는 캐시(1시간 TTL)를 적용하여 반복 조회 부하를 줄인다.

## 3. 정상작동 시나리오

### 시나리오 1: 정산 조회 → 지급 완료

| 단계 | 사용자 행동 | 시스템 응답 | 검증 포인트 |
| :---: | :--- | :--- | :--- |
| 1 | 정산 목록 진입 | 당월 정산 목록 로드 | 기본 필터=당월 |
| 2 | 매장 "강남점" 필터 적용 | 해당 매장 정산 목록만 표시 | 검색 가능 Select |
| 3 | "정산완료" 행 클릭 | 상세 페이지 이동 | 정산 구성요소 표 렌더링 |
| 4 | 매출/배달비/할인/수수료/실정산액 확인 | 각 항목 천단위 콤마 + ₩ 표기 | netAmount 볼드 강조 |
| 5 | 하단 주문별 상세 테이블 확인 | 개별 주문 목록 렌더링 | 주문번호, 금액, 결제수단 |
| 6 | 정산완료 상태 확인 | 정산 배치 실행 후 자동 전환됨 | status=completed, paymentDate 자동 기록 |

### 시나리오 2: 통계 조회

## 4. 개발자용 정책 설명

### 4.1. 정산액 계산 공식

```
■ 공식
netAmount = totalSales + deliveryFee + hqSupport - promotionDiscount - platformFee - pgFee

■ 각 항목 설명 (실무 용어)
  totalSales (매출 합계)       : 정산 기간 내 완료 주문의 상품 금액 합계
  deliveryFee (배달비 합계)     : 배달 주문에서 고객이 낸 배달비 합계
  hqSupport (본사 지원금)       : 본사가 부담하는 할인금 (가맹점에 보전)
                                예) 쿠폰 10,000원 할인, 본사 70% 부담 → 7,000원
  promotionDiscount (할인 총액) : 아래 3가지의 합계
    - couponAmount : 쿠폰으로 할인된 금액
    - pointUsed    : 적립 포인트로 결제한 금액
    - discountAmount: 프로모션/이벤트 자동 할인 금액
  platformFee (플랫폼 수수료)   : 매출액 × 플랫폼 수수료율 (플랫폼 운영 수익)
                                수수료율 = super-admin FeePolicy.baseRate
  pgFee (PG사 결제 수수료)      : 매출액 × PG 수수료율 (결제대행사 비용)
                                수수료율 = super-admin FeePolicy.pgFeeRate
                                브랜드별로 다를 수 있음 (PG사 계약 조건에 따라)

■ 음수 정산에 대해
  플랫폼 수수료(매출×%)와 PG 수수료(매출×%)는 정률 구조이므로,
  수수료가 매출을 초과하여 음수가 되는 일은 구조적으로 발생하지 않는다.
  예외: 이전 정산 확정 후 취소/환불 차감액이 당기 매출보다 클 때 음수 가능.
  시스템에서 사전 차단하지 않으며, 발생 시 운영자 알림 + 빨간색 강조 표시.
```

### 4.2. 정산 상태 정의

```
■ 상태값 (2가지)
  pending   : 정산전 — 정산일이 아직 도래하지 않은 상태
  completed : 정산완료 — 정산 배치가 실행되어 정산이 확정된 상태

■ 전이 규칙
  pending → completed : 정산 배치 실행 시 자동 전환 (정산일 익일 새벽 02:00)

■ 정산일 기준
  반월 단위(1~15일 / 16~말일) 종료 시점이 정산 기준일이다.
  정산 기준일 이전: pending (정산전)
  정산 배치 실행 후: completed (정산완료)
```

### 4.3. 정산 마감 및 취소/환불 정책

```
■ 정산 주기 및 배치 실행 시점
  · 정산 주기: 반월 단위 (1일~15일 / 16일~말일)
  · 배치 실행: 기간 종료 익일 새벽 02:00 자동 실행
    예) 3월 1~15일 정산 → 3월 16일 새벽 02:00에 배치 실행

■ 정산 마감 기준 (스냅샷 시점)
  정산은 "영업일 기준"으로 마감한다.
  스냅샷 시점 = 기간 종료일의 영업 마감 시각 (00:00, 자정)
  배치가 새벽 02:00에 실행되더라도, 데이터 기준은 자정(00:00)이다.

■ 미완료 주문 자동 완료 처리
  영업일 기준 자정(00:00)이 지나면, 해당 영업일의 미완료 주문
  (pending/confirmed/preparing/ready 상태)을 자동으로 completed(주문완료)로 변경한다.
  이후 정산 배치가 실행되어 정산을 진행한다.

  처리 순서:
  1) 자정(00:00) — 전일 미완료 주문을 주문완료(completed)로 자동 전환
  2) 새벽 02:00 — 정산 배치 실행 (completed 주문만 집계)

  예시)
  · 3월 15일 22:00에 접수된 주문이 배달 완료 처리 없이 자정을 넘김
  → 3월 16일 00:00에 자동으로 completed 처리
  → 3월 16일 02:00 정산 배치에서 정상 집계

  ※ cancelled(취소) 주문은 자동 완료 대상이 아니다.
  ※ payment_pending(결제 대기), expired(결제 만료) 주문도 대상이 아니다.

■ 배치 실행 전 취소 요청이 들어오면?
  배치 실행 시점(새벽 02:00)이 아닌, 스냅샷 시점(전일 23:59:59)이 기준이다.
  따라서 배치 실행 전이라도 스냅샷 이후 취소는 당기 정산에 반영되지 않는다.

  예시)
  · 정산 기간: 3월 1~15일
  · 스냅샷 시점: 3월 15일 23:59:59
  · 배치 실행: 3월 16일 새벽 02:00
  · 고객이 3월 16일 새벽 01:30에 취소 요청
  → 스냅샷(3/15 23:59:59) 이후이므로 당기 정산에 미반영
  → 해당 취소는 3월 16~31일 정산에서 차감 처리

■ 정산 후 취소
  주문은 정산 여부와 관계없이 언제든 고객 요청 시 취소 가능하다.
  시스템에서 취소를 차단하지 않는다.

■ 마감 후 취소 시 마이너스 주문내역 생성
  정산 완료 후 취소가 발생하면:
  1) 이미 마감된 정산 건은 소급 수정하지 않는다.
  2) 취소된 주문과 동일한 주문번호로 "마이너스 주문내역"을 신규 생성한다.
     · 주문번호: 원본과 동일 (예: 20260310-000042)
     · 금액: 원본 금액의 음수 (예: -15,000원)
     · 유형: 'refund' (환불/취소 차감)
     · 참조: 원본 주문 ID 연결
  3) 마이너스 주문내역은 다음 정산 기간에 자동 집계되어 차감 반영된다.

  예시)
  · 3월 1~15일 정산이 완료(정산완료)된 상태
  · 3월 10일 주문(15,000원)이 3월 18일에 취소 처리됨
  → 3월 1~15일 정산은 그대로 유지
  → 주문번호 20260310-000042로 마이너스 내역(-15,000원) 생성
  → 해당 마이너스 내역은 3월 16~31일 정산에서 차감

■ 정산 배치 실행 중 주문 변경
  배치가 돌고 있는 도중에 주문 취소가 발생해도,
  스냅샷 시점(기간 종료일 23:59:59) 기준으로 산출하므로 영향 없다.
  해당 취소는 다음 정산에 자동 반영된다.

■ 핵심 요약
  · 정산 후 취소: 언제든 가능 (시스템 차단 없음)
  · 취소 시: 동일 주문번호의 마이너스 주문내역이 생성됨
  · 마이너스 내역은 다음 정산에서 차감 처리
  · 원본 정산은 소급 수정하지 않음
```

### 4.4. 본사 지원금(hqSupport) — 본사가 대신 내주는 할인금

```
■ 계산 방식
  프로모션별: 할인 금액 × (본사 부담 비율 / 100)

■ 쉬운 설명
  고객이 쿠폰이나 프로모션으로 할인을 받으면, 그 할인 금액을
  "본사"와 "가맹점"이 나눠서 부담한다.
  본사가 부담하는 부분(hqSupport)은 가맹점에 보전해주므로,
  가맹점 입장에서는 수입으로 합산된다.

■ 예시
  · 이벤트 할인: 10,000원
  · 본사 부담 비율: 70%
  → 본사 지원금(hqSupport) = 7,000원 (본사가 가맹점에 보전)
  → 가맹점 부담 = 3,000원 (가맹점 정산에서 차감)

■ 왜 중요한가?
  본사 지원금 비율이 낮을수록 가맹점 부담이 커지고,
  과도한 할인 이벤트 시 가맹점 정산액이 줄어들 수 있으므로
  프로모션 설정 시 정산 영향을 반드시 확인해야 한다.
```

### 4.5. E쿠폰(교환권/금액권) 정산 — 정산주체는 외부 쿠폰사

```
■ E쿠폰이란?
  고객이 외부 쿠폰사(기프티콘, 카카오선물 등)에서 발급받은
  교환권(exchange) 또는 금액권(voucher)으로 결제하는 방식이다.

■ 정산 주체
  E쿠폰 결제 금액은 플랫폼이나 가맹점이 직접 정산하지 않는다.
  외부 쿠폰사(ECouponUsage.couponCompany)가 정산 주체이며,
  플랫폼 ↔ 쿠폰사 간 별도 정산 계약에 따라 처리된다.

■ 정산 처리 방식
  · 교환권(exchange): 특정 상품과 1:1 교환. 쿠폰사가 해당 상품 금액 전액 정산
  · 금액권(voucher): 금액 차감 방식. 쿠폰사가 사용 금액만큼 정산
  · 가맹점 정산에서 vouchersUsed는 "외부 쿠폰사 정산 예정 금액"으로 표기
  · 가맹점 실정산액(netAmount)에는 직접 영향을 주지 않음

■ UI 표시
  정산 상세 화면에서 "E쿠폰 사용액" 항목은 참고용으로 표시하되,
  "정산주체: 외부 쿠폰사 (○○○)" 형태로 쿠폰사명을 병기한다.
```

### 4.6. 포인트 사용 시 본사/가맹점 부담 비율

```
■ 설정 위치
  마케팅관리 > 포인트 설정 > 사용 정책 > "포인트 사용 시 정산 부담 비율"
  (PointSettings.usePolicy.headquartersRatio / franchiseRatio)

■ 동작 방식
  고객이 포인트로 결제하면, 해당 금액을 본사와 가맹점이
  설정된 비율로 나눠서 부담한다.
  두 비율의 합은 반드시 100%여야 한다.

■ 예시
  · 포인트 부담 비율: 본사 70% / 가맹점 30%
  · 고객이 1,000P 사용
  → 본사 부담: 700원 (가맹점에 보전)
  → 가맹점 부담: 300원 (정산에서 차감)

■ 정산 반영
  pointsUsed 금액 중 (headquartersRatio / 100) 비율은 hqSupport에 합산되어
  가맹점 수입으로 보전된다.
  나머지 (franchiseRatio / 100) 비율은 가맹점 부담으로 정산에서 차감된다.

■ 쿠폰과의 차이
  쿠폰: 쿠폰별로 개별 부담 비율 설정 (쿠폰 생성 시)
  포인트: 전역 설정으로 일괄 적용 (포인트 정책에서 관리)
```

## 정산 (Settlement) API

### 9-1. 매장별 월별 정산 내역 조회
```
GET /settlements/monthly
```
**Query Parameters**
- `yearMonth` (string, Y) - 예: `2026-03`
- `storeId` (string, N)
- `page`, `limit`

**Response** `200 OK`
```json
{
  "data": [
    {
      "storeId": "store-1",
      "totalSales": 15000000,
      "headquartersFee": 1500000,
      "couponFranchiseBurden": 200000,
      "finalSettlementAmount": 13300000
    }
  ],
  "pagination": { ... }
}
```

### 9-2. 정산 엑셀 리포트 다운로드
```
GET /settlements/monthly/export
```
**Response** `200 OK` (Excel 파일 바이너리 스트림)
