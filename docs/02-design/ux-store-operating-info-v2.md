## 변경 1: 매장 뱃지 제거

### 현재 상태 (AS-IS)

노출설정 모달(`VisibilitySettingsEditModal`)에 다음 항목이 존재:
- 신규 뱃지 표시 여부 + 종료일
- 이벤트 뱃지 표시 여부 + 뱃지 문구

### 변경 후 (TO-BE)

위 4개 필드(신규 뱃지, 뱃지 종료일, 이벤트 뱃지, 뱃지 문구)를 **화면에서 완전 제거**.

### 사용자 흐름

| 단계 | 설명 |
|------|------|
| 진입 | 매장 상세 > 노출설정 수정 모달 열기 |
| 화면 | 뱃지 관련 영역이 보이지 않음 (검색 설정, 추천 매장만 노출) |
| 완료 | 기존과 동일하게 저장 |

### UI 상태

| 상태 | 처리 방식 |
|------|----------|
| 기존 데이터에 뱃지 값이 있는 매장 | 화면에 표시하지 않음. 저장 시 해당 필드를 전송하지 않아 서버에서 자연 소멸 |
| 신규 매장 | 뱃지 필드 자체가 없으므로 변화 없음 |

### 주의사항

- **마이그레이션 안내 불필요**: 관리자가 직접 뱃지를 끄는 작업이 필요 없음. 화면에서 사라지면 끝.
- **잔존 데이터**: 서버 측에서 기존 뱃지 데이터를 정리할 시점(일괄 삭제 또는 자연 만료) 결정 필요 -- 백엔드 협의 사항.

## 변경 2: 단일 노출 설정

### 현재 상태 (AS-IS)

노출설정 모달에서 6개 채널(자사 앱, 웹, 키오스크, 배달의민족, 요기요, 쿠팡이츠)별로 개별 켜기/끄기 체크박스 제공.

### 변경 후 (TO-BE)

채널별 노출을 **단일 켜기/끄기 토글(isVisible)** 하나로 통합.
- 켜면 웹/모바일 모든 채널에 노출
- 끄면 모든 채널에서 비노출

기존 유지 항목: 검색 노출(isSearchable), 추천 매장(isRecommended), 추천 순서(recommendedOrder)

### 사용자 흐름

| 단계 | 설명 |
|------|------|
| 진입 | 매장 상세 > 노출설정 수정 모달 열기 |
| 화면 | 채널별 체크박스 영역이 사라지고, **"매장 노출" 단일 토글**이 최상단에 표시 |
| 조작 | 토글 끄기 시 확인 대화상자: "매장을 비노출하면 모든 채널에서 고객에게 보이지 않습니다. 비노출하시겠습니까?" |
| 저장 | 변경된 isVisible 값 전송 |

### UI 상태

| 상태 | 화면 표현 |
|------|----------|
| 노출 켜짐 | 토글 ON + "노출중" 텍스트 (초록색) |
| 노출 꺼짐 | 토글 OFF + "비노출" 텍스트 (회색) + 하위 설정(검색, 추천) 비활성화(흐리게 처리) |
| 로딩 중 | 토글 비활성화 + 작은 회전 아이콘 |
| 저장 실패 | 토글 원래 값으로 복원 + 오류 알림: "노출설정 변경에 실패했습니다" |

### 인터랙션 상세

1. **비노출 시 하위 설정 연동**: isVisible이 꺼지면 isSearchable, isRecommended도 UI에서 비활성화(회색 처리). 단, 값 자체는 유지하여 다시 켜면 복원됨.
2. **기존 채널별 데이터 마이그레이션**: 6개 채널 중 1개라도 켜져 있으면 isVisible = true로 변환. 모두 꺼져 있으면 isVisible = false.

## 변경 3: 최소주문금액 매장관리로 이동

### 현재 상태 (AS-IS)

- **배달 최소주문금액**: 읽기 전용으로 표시 + "상권관리에서 설정" 링크 제공 (상권관리 페이지로 이동해야 수정 가능)
- **포장 최소주문금액**: 포장 설정 영역 내 입력 필드로 직접 수정 가능

### 변경 후 (TO-BE)

- **배달 최소주문금액**: 영업정보 수정 페이지 내 배달 설정 카드에서 **직접 편집 가능한 숫자 입력 필드**로 변경
- **포장 최소주문금액**: 기존과 동일하게 직접 편집 유지
- "상권관리에서 설정합니다" 안내 문구 및 링크 **제거**
- "배달비는 상권관리에서 설정합니다" 안내 및 이동 링크는 **유지** (배달비는 여전히 상권관리 소관)

### 사용자 흐름

| 단계 | 설명 |
|------|------|
| 진입 | 매장 상세 > 영업정보 수정 > 배달 설정 카드 |
| 화면 | "최소 주문금액" 입력 필드가 직접 수정 가능 상태로 표시 |
| 조작 | 금액 입력 (숫자만) |
| 검증 | 0원 이상 정수만 허용, 음수/문자 입력 시 입력 차단 |
| 저장 | 배달 설정 내 minOrderAmount로 서버 전송 |

### UI 상태

| 상태 | 화면 표현 |
|------|----------|
| 배달 켜짐 | 최소주문금액 입력 필드 활성화 |
| 배달 꺼짐 | 배달 설정 하위 항목 전체 숨김 (기존과 동일) |
| 미입력 | 빈칸 허용 -- 빈칸이면 "제한 없음"으로 해석 (서버 기본값 0원) |
| 입력 오류 | 필드 하단 빨간색 안내 "숫자만 입력할 수 있습니다" |
| 저장 성공 | 성공 알림 + 상세 페이지로 복귀 |
| 저장 실패 | 오류 알림 + 폼 유지 |

### 인터랙션 상세

1. **금액 입력 형식**: 천 단위 콤마 자동 표시 (예: 15,000), 저장 시 숫자값으로 변환
2. **단위 표시**: 입력 필드 오른쪽에 "원" 표기
3. **배달비 안내 유지**: 배달비는 상권관리에서 설정하므로 "배달비는 상권관리에서 설정합니다 > 상권관리로 이동" 안내는 그대로 유지

## 변경 4: 영업정보 UI 박스 분리

### 현재 상태 (AS-IS)

영업정보 수정 페이지(`OperatingInfoEdit.tsx`)는 하나의 카드 안에 아코디언(접기/펼치기)으로 구성:
- 매장 노출 / 영업시간 / 배달 설정 / 포장 설정 / 예약 설정 / 임시 휴업

하나의 `<form>`으로 전체를 감싸고, "저장" 버튼 1개로 모든 항목을 한 번에 저장.

### 변경 후 (TO-BE)

5개 독립 카드(네모 상자)로 분리하되, **저장은 여전히 한 번에** 처리.

> 이유: 카드별 독립 저장으로 바꾸면 API 5개 호출이 필요하고, 배달/포장 연동 검증이 카드 간 분리되어 복잡해짐. 한 번에 저장하되 시각적 그룹핑만 분리하는 것이 적절.

### 5개 카드 구성

#### 카드 1: 운영 상태

| 항목 | 설명 |
|------|------|
| 앱 운영 상태 | 현재 앱에 표시되는 상태 선택 (영업중/준비중/휴식시간/영업종료/임시휴업) -- 드롭다운 |
| 임시휴업 | 켜기/끄기 토글 + 사유 선택(재료 소진, 기기점검, 개인사정, 기타) + 기간 |

**인터랙션**:
- 앱 운영 상태에서 "임시휴업" 선택 시 -> 임시휴업 토글 자동 ON + 사유/기간 입력 영역 펼침
- 임시휴업 토글 ON 시 -> 앱 운영 상태가 "임시휴업"으로 자동 변경
- 임시휴업 토글 OFF 시 -> 앱 운영 상태가 직전 상태(open)로 복원

#### 카드 2: 영업시간

| 항목 | 설명 |
|------|------|
| 설정 방식 | 간편 설정(평일/주말) 또는 요일별 설정 -- 라디오 버튼 |
| 평일/주말/공휴일 | 각 구간별 영업 여부 토글 + 영업시간 + 라스트오더 + 휴게시간 |
| 요일별 | 월~일 각각 영업 여부 토글 + 영업시간 + 라스트오더 + 휴게시간 |

**인터랙션**:
- 간편/요일별 전환 시 기존 입력 데이터 유지 (요일별로 전환해도 평일 데이터가 월~금에 자동 복사)
- 요일별에서 간편으로 돌아올 때는 요일별 데이터 유지 (저장 시 활성 모드 기준으로 전송)

#### 카드 3: 배달 설정

| 항목 | 설명 |
|------|------|
| 배달 가능 | 켜기/끄기 토글 |
| 최소주문금액 | 숫자 입력 필드 (원) -- **직접 편집 가능 (변경 3 반영)** |
| 예상 배달시간 | 숫자 입력 필드 (분) -- 고객 앱 안내용 |
| 배달비 안내 | "배달비는 상권관리에서 설정합니다 > 상권관리로 이동" (읽기 전용 안내) |

#### 카드 4: 포장 설정

| 항목 | 설명 |
|------|------|
| 포장 가능 | 켜기/끄기 토글 |
| 최소주문금액 | 숫자 입력 필드 (원) |
| 예상 준비시간 | 숫자 입력 필드 (분) -- 고객 앱 안내용 |

#### 카드 5: 예약 설정

| 항목 | 설명 |
|------|------|
| 예약 주문 | 켜기/끄기 토글 + "배달/포장 모두에 적용됩니다" 안내 |
| 예약 가능 시간 | "현재 시간 기준 [N]분 후부터 예약 가능" 숫자 입력 |

**인터랙션**:
- 배달과 포장이 모두 꺼져 있으면 예약 토글 비활성화 + 안내: "배달 또는 포장을 먼저 켜주세요"

### 사용자 흐름 (전체)

```
매장 상세 페이지
  └─ "영업정보 수정" 버튼 클릭
      └─ 영업정보 수정 페이지 진입
          ├─ 5개 카드가 세로로 나열 (모두 펼쳐진 상태)
          ├─ 각 카드 내용 수정
          ├─ 하단 고정 바: [취소] [저장]
          └─ 저장 클릭
              ├─ 검증 통과 -> 성공 알림 + 매장 상세로 이동
              └─ 검증 실패 -> 해당 카드로 자동 스크롤 + 오류 표시
```

### UI 상태 (공통)

| 상태 | 화면 표현 |
|------|----------|
| 로딩 | 전체 페이지 중앙에 회전 아이콘 (카드 구조 미노출) |
| 데이터 로드 완료 | 5개 카드 표시, 기존 값 채워짐 |
| 빈 값 (신규 매장) | 각 필드에 기본값 적용 (배달/포장 ON, 영업시간 11:00~22:00) |
| 저장 중 | 저장 버튼 비활성화 + 회전 아이콘 + "저장 중..." |
| 저장 성공 | 성공 알림: "영업정보가 수정되었습니다" + 매장 상세로 복귀 |
| 저장 실패 | 오류 알림: "영업정보 수정에 실패했습니다" + 폼 유지 (사용자 입력 보존) |
| 변경 없이 저장 | 안내 알림: "변경사항이 없습니다" + 매장 상세로 복귀 |
| 뒤로 가기 (변경사항 있을 때) | 확인 대화상자: "저장하지 않은 변경사항이 있습니다. 페이지를 떠나시겠습니까?" |

## 카드 간 연동 규칙

### 배달/포장 상호 검증 (최소 1개 필수)

```
[배달 끄기 시도]
  ├─ 포장이 켜져 있음 -> 배달 끄기 허용
  └─ 포장이 꺼져 있음 -> 배달 끄기 차단 + 알림: "배달 또는 포장 중 최소 1개는 활성화해야 합니다"

[포장 끄기 시도]
  ├─ 배달이 켜져 있음 -> 포장 끄기 허용
  └─ 배달이 꺼져 있음 -> 포장 끄기 차단 + 동일 알림
```

- 이 검증은 **토글 조작 시점**(즉시)과 **저장 시점**(이중 확인) 모두 적용

### 예약 <-> 배달/포장 연동

| 배달 | 포장 | 예약 토글 |
|------|------|----------|
| ON | ON | 활성화 |
| ON | OFF | 활성화 |
| OFF | ON | 활성화 |
| OFF | OFF | 비활성화 + "배달 또는 포장을 먼저 켜주세요" 안내 |

- 배달/포장이 둘 다 꺼지면 예약도 **자동으로 OFF** + 알림: "배달/포장이 모두 비활성화되어 예약도 함께 꺼졌습니다"

### 임시휴업 <-> 앱 운영 상태 연동

| 임시휴업 토글 | 앱 운영 상태 |
|-------------|------------|
| ON | "임시휴업"으로 자동 변경 |
| OFF | 직전 상태(기본: "영업중")로 복원 |

- 앱 운영 상태 드롭다운에서 "임시휴업" 선택 시 임시휴업 토글도 자동 ON

### 카드 간 연동 시각적 표현

- 연동으로 인해 다른 카드의 값이 변경되었을 때: 해당 카드에 **잠깐 깜빡이는 파란색 테두리 효과** (1.5초) + 변경된 항목 옆에 "자동 변경됨" 작은 텍스트 표시 (3초 후 사라짐)
- 이유: 사용자가 자신이 조작하지 않은 카드의 값이 바뀌었음을 인지할 수 있어야 함

## 접근성 고려사항

### 키보드 탐색

| 요소 | 키보드 동작 |
|------|-----------|
| 토글(켜기/끄기) | Tab으로 포커스 이동, Space로 켜기/끄기 전환 |
| 시간 입력 | Tab으로 포커스, 직접 타이핑 또는 위/아래 화살표 |
| 라디오(간편/요일별) | Tab으로 그룹 진입, 좌우 화살표로 선택 전환 |
| 휴업 사유 버튼 | Tab으로 순차 이동, Enter/Space로 선택 |
| 저장/취소 | Tab으로 포커스, Enter로 실행 |

### 화면 읽기 보조 (스크린리더)

| 요소 | 보조 기술 지원 |
|------|-------------|
| 토글 | `role="switch"` + `aria-checked` + 라벨 연결 (예: "배달 가능 토글, 현재 켜짐") |
| 카드 | `role="region"` + `aria-labelledby`로 카드 제목과 연결 |
| 오류 메시지 | `role="alert"` + `aria-live="assertive"` (오류 발생 즉시 읽어줌) |
| 성공 알림 | `aria-live="polite"` (현재 읽기가 끝난 후 읽어줌) |
| 비활성 필드 | `aria-disabled="true"` + 비활성 사유 설명 (예: "배달이 꺼져 있어 입력할 수 없습니다") |

### 색상 대비

- 토글 ON(초록): 배경과 최소 4.5:1 대비 비율 확보
- 토글 OFF(회색): "꺼짐" 텍스트 라벨 병기 (색상만으로 상태 구분하지 않음)
- 오류 텍스트(빨강): 아이콘(!) 병기 (색각 이상 사용자 대응)
- 비활성 상태: 투명도 60% + "비활성" 텍스트 표시

### 터치 영역

- 모든 토글/버튼: 최소 44x44px 터치 영역 확보
- 시간 입력 필드: 높이 48px (h-12) 유지

## UX 보완 제안

### 제안 1: 변경사항 이탈 방지

**현재 문제**: 사용자가 여러 카드를 수정한 뒤 실수로 뒤로 가기를 누르면 모든 변경 내용이 사라짐.

**제안**: 페이지 이탈 시 확인 대화상자 표시.
- `beforeunload` 이벤트로 브라우저 이탈 방지
- React Router의 `useBlocker`로 앱 내 이동 차단
- 대화상자: "저장하지 않은 변경사항이 있습니다. 페이지를 떠나시겠습니까?" [머무르기 / 나가기]

### 제안 2: 카드별 변경 표시기

**현재 문제**: 5개 카드 중 어떤 카드를 수정했는지 한눈에 알기 어려움.

**제안**: 수정된 카드 제목 옆에 파란색 점(indicator) 표시.
- 카드 내 어떤 값이든 초기값과 달라지면 제목 옆에 파란 점 표시
- 저장 완료 시 파란 점 제거

### 제안 3: 배달 최소주문금액 변경 이력 안내

**현재 문제**: 배달 최소주문금액이 상권관리에서 매장관리로 이동하면서, 기존에 상권관리에서 설정된 값과의 정합성 문제 발생 가능.

**제안**:
- 마이그레이션 시 상권관리의 기존 값을 매장 배달 설정에 자동 복사
- 최초 1회, 배달 최소주문금액 필드 상단에 안내 표시: "기존 상권관리에서 설정된 금액이 적용되어 있습니다. 이 페이지에서 직접 수정할 수 있습니다." (닫기 가능한 안내 배너)

### 제안 4: 5개 카드 레이아웃 최적화

**현재 문제**: 5개 카드가 세로로 길게 나열되면 스크롤이 많아짐.

**제안**:
- 데스크톱(1280px 이상): 2열 배치
  - 왼쪽 열: 운영 상태 + 영업시간 (세로 길이가 큰 항목)
  - 오른쪽 열: 배달 설정 + 포장 설정 + 예약 설정
- 태블릿/모바일(1280px 미만): 1열 세로 나열 (현재와 동일)
- 하단 저장 버튼 바: 화면 하단 고정 (스크롤과 무관하게 항상 보임)

### 제안 5: 저장 전 변경 요약

**현재 문제**: 사용자가 여러 카드를 수정하면 정확히 무엇을 변경했는지 최종 확인이 어려움.

**제안**: 저장 버튼 클릭 시 변경 요약 대화상자 표시.
- 변경된 항목만 "이전 값 -> 새 값" 형태로 나열
- 예시:
  ```
  변경 내용:
  - 배달 최소주문금액: 15,000원 -> 12,000원
  - 포장 가능: 꺼짐 -> 켜짐
  - 예상 배달시간: 30분 -> 40분
  ```
- [취소] [저장] 버튼

> 참고: 이 제안은 변경 항목이 1개일 때는 건너뛰고 바로 저장해도 무방. 2개 이상 변경 시에만 표시하는 것을 권장.

### 제안 6: 앱 운영 상태를 운영 상태 카드에 추가

**현재 문제**: PRD에서 "운영 상태" 카드에 앱 운영 상태 토글을 포함한다고 했으나, 현재 `OperatingInfoEdit`에는 `appOperatingStatus` 드롭다운이 없음. 타입에는 정의되어 있지만(`AppOperatingStatus`) UI에서 누락.

**제안**: 운영 상태 카드 최상단에 앱 운영 상태 드롭다운 추가.
- 선택지: 영업중 / 준비중 / 휴식시간 / 영업종료 / 임시휴업
- 임시휴업 선택 시 하단 임시휴업 영역 자동 펼침 (위 연동 규칙 참고)

### 제안 7: 노출설정 모달 -> 영업정보 페이지 통합 검토

**현재 문제**: "매장 노출(isVisible)"이 영업정보 수정 페이지와 노출설정 모달 양쪽에 존재. 변경 2로 채널별 노출이 단일 토글로 바뀌면, 노출설정 모달의 역할이 크게 줄어듦 (isVisible + isSearchable + isRecommended만 남음).

**제안**: 두 가지 선택지 중 결정 필요.
- **A안**: 노출설정 모달 유지 (isVisible, isSearchable, isRecommended 3개 항목). 영업정보의 isVisible은 제거하여 중복 방지.
- **B안**: 노출설정 모달 폐지. isSearchable, isRecommended를 영업정보 수정 페이지에 "노출 설정" 카드(6번째)로 통합.
- **권장**: A안 (모달 유지). 노출 설정은 영업정보와 성격이 다르고, 매장 상세에서 빠르게 토글하는 용도로 모달이 더 적합.
