# 관리자 대시보드 로그인 페이지 개발 문서

## 개요
이 문서는 관리자 대시보드 시스템의 인증 진입점인 로그인 페이지(`LoginPage.tsx`)에 대한 세부 구현 명세입니다. 기능 단위별 로직과 구조를 개발자가 명확하게 파악하고 유지보수할 수 있도록 다나까체로 상세히 구성하였습니다. 로그인 페이지는 로그인 폼, 2차 인증(MFA) 처리, 계정 잠금 카운트다운 방어 로직 등을 하나의 컴포넌트 내에서 동적으로 렌더링하도록 구현되어 있습니다.

---

## 1. 전역 상태 및 라우팅 제어 (State & Routing)

로그인 페이지는 주로 Zustand 기반의 `useAuth` 훅을 활용해 전역 상태를 추적 및 갱신합니다.

### 1-1. 전역 상태 연동
`useAuth` 훅에서 다음 상태와 메서드를 가져와 사용합니다:
- **`isAuthenticated`**: 완전히 인증이 완료되었는지 확인합니다.
- **`isMfaRequired`**: 1차 이메일/비밀번호 인증은 통과하였으나 2차 인증이 필요한지 판별합니다.
- **`isLoading`**: 비동기 API 통신(로그인, MFA 검증 등) 중일 때 스피너 등을 활성화합니다.
- **`errorMessage`**: 인증 실패 시 발생한 에러를 표시합니다.
- **`login`, `verifyMfa`, `clearError`**: 인증 요청 및 상태 초기화를 위한 내부 메서드입니다.
- **`pendingMfaUserId`**: 1차 인증 통과 후, MFA 재요청 등을 처리하기 위한 유저의 식별키를 `useAuthStore`에서 직접 가져와 사용합니다.

### 1-2. 자동 리다이렉션 (Routing)
최상단 렌더링 라이프사이클(`useEffect`)에서 `isAuthenticated` 상태를 감지하여 인증 완료 시 즉시 라우팅을 처리합니다.
- 사용자가 권한 없이 딥링크(예: `/orders`)를 통해 접근했다가 로그인 화면으로 강제 이동된 이력이 있는 경우, `location.state.from.pathname`을 추적하여 원래 가려던 페이지로 이동시킵니다.
- 별도의 이력이 없다면 기본 메인 페이지인 `/dashboard` 경로로 자동 이동(replace) 처리합니다.

## 2. 보안 기능 및 시도 제한 (Security & Lockout)

비정상적인 접근 시도 방어를 위해 로컬 컴포넌트 상태를 통해 실시간 계정 잠금 및 알림 처리를 구현하였습니다.

### 2-1. 계정 잠금 로직 관리
- **연속 실패 처리**: 사용자가 최초 이메일을 폼력 시, `authService.getLoginAttemptInfo`를 호출해 해당 ID(이메일)의 잔여 로그인 시도 횟수 및 잠금 상태 정보를 불러옵니다. 이 정보는 `attemptInfo` `useState`에 저장됩니다.
- **카운트다운(`lockCountdown`)**: 계정이 잠긴 경우(`isLocked: true`), `lockedUntil` 만료 시간을 기준으로 `setInterval`을 호출해 실시간으로 1초마다 남은 잠금 해제 시간을 갱신(&#x60;setLockCountdown&#x60;)합니다. 시간이 모두 경과하면 `isLocked` 상태를 `false`로 변환하여 입력 필드의 비활성화 상태를 해제합니다.
- 사용자는 잠긴 시간 동안 어떠한 폼 조작이나 Submit 이벤트를 발생시킬 수 없습니다. (입력 필드 및 버튼 `disabled` 처리)

## 3. 기능 단위 세부 동작

로그인 페이지는 `isMfaRequired` 상태를 플래그로 삼아 2차 인증 화면과 기본 로그인 화면 중 적절한 사용자 경험을 분기 렌더링합니다.

### 3-1. 1차 인증: 기본 로그인 (LoginForm)
이메일/아이디와 비밀번호를 검증하는 가장 기본적인 Form 영역입니다.

- **폼 제출 제어**: 요소 내 `form`에서 `handleSubmit`이 작동하면 화면 상의 기존 에러(`clearError()`)를 제거한 후 상태의 `email`, `password`로 `login` 함수를 콜백합니다.
- 로그인 실패 판정 시(`success: false`), 곧바로 `getLoginAttemptInfo`를 호출하여 재차 계정 상태(실패 횟수 누적 및 잠금 여부 등)를 업데이트하고, 상태를 반영하여 잠금 UI를 트리거합니다.

### 3-2. 2차 인증: 이메일 검증 (MFA Screen)
1차 폼에서 인증이 성공했으나 2차 인증(MFA)이 강제된 계정일 때 나타납니다.

- **OTP 입력 처리**: 커스텀 `OTPInput` 컴포넌트롤 통해 6자리의 인증 코드를 입력받습니다. 모든 자리의 코드가 정상적으로 채워지면 `onComplete` 이벤트로 `handleOtpComplete` 로직을 자동으로 구동합니다.
- **인증 코드 재발송 시스템**:
  - `pendingMfaUserId`를 기반으로 등록된 이메일을 향해 인증 코드를 재전송합니다(`authService.resendVerificationCode()`).
  - 재전송 스팸 방지를 위해 `resendCooldown` 상태를 두어 버튼 클릭 시 60초의 쿨다운(`setInterval`)을 강제합니다 제약 시간 중에 요청 버튼은 비활성화됩니다.
- 재전송을 요청하면 이메일 일부가 마스킹(`setMaskedEmail()`)된 문자열이 반환되며, 이를 페이지 상단에 안내 모달 텍스트로 띄워 사용자가 혼선을 겪지 않게 돕습니다.

## 4. UI 렌더링 및 디버깅 구조 (UI & Guides)

### 4-1. 디자인 컴포넌트 설계
- 모든 정보 입력 창은 공용 디자인 시스템에 등록된 `Button`, `Input`, `Label` 등의 모듈화된 UI를 가져다 사용하여 전역 디자인을 통일하였습니다.
- 화면은 `login-container` 클래스를 기반으로 Flexbox 가운데 정렬이 적용된 `login-card`에 집중 배치되어 있습니다. 사용자 시선의 분산을 막고 인증 자체에만 목적을 두도록 설계되어 있습니다.

### 4-2. 개발 서버(Dev) 환경 편의 제공
코드는 Vite 개발 서버에서 구동될 때만 한정적으로 동작하는 정보들을 포함합니다 (`import.meta.env.DEV` 조건).

- **테스트 계정 가이드 블록표시**: 실 서버에서는 보이지 않는 개발용 하드코딩된 더미 계정 목록들을 보여주어 QA 혹은 시나리오 테스트 과정에서 개발자가 정보를 즉시 확인할 수 있도록 배치하였습니다.
- **DevGuide 툴팁**: 우측 상단 플로팅 팝업인 `DevGuide` 컴포넌트를 사용해 개발 및 테스트 중 API 연동 가이드나 기획 의도 등 모의(Mock) 명세 정보를 지속적으로 설명해 줍니다. (`LOGIN_DEV_GUIDE` 활용)

## 5. 인증 에러 코드 (Error & Validation)

인증 흐름 중 발생할 수 있는 여러 계정 제한과 상태 이상은 명확한 에러 코드로 파악할 수 있도록 분류되어 있습니다. `src/types/auth.ts` 내부의 `AUTH_ERROR_MESSAGES` 상수에 정의되어 있으며, 개발자는 API 연동 시 백엔드 응답(Error Code)이 아래와 완벽히 매핑되는지 점검해야 합니다. 

### 5-1. 계정 및 비밀번호 검증 타입 에러
| 에러 코드 (`AuthErrorCode`) | 실제 출력 텍스트 | 발생 조건 및 상세 흐름 |
| --- | --- | --- |
| `INVALID_CREDENTIALS` | "이메일 또는 비밀번호가 올바르지 않습니다." | 시스템에 등록되지 않은 이메일이거나 비밀번호가 불일치할 때 발생합니다. |
| `ACCOUNT_LOCKED` | "계정이 잠겼습니다. 관리자에게 문의하세요." | 비밀번호 오류 시도 한도(5회)를 연속적으로 초과하여 계정이 일시적 혹은 영구적으로 잠금 처리되었을 때 반환합니다. |
| `ACCOUNT_INACTIVE` | "비활성화된 계정입니다." | 퇴사 혹은 특정 사유로 관리자에 의해 계정 사용이 정지된 상태일 경우 발생합니다. |
| `ACCOUNT_PENDING` | "승인 대기 중인 계정입니다." | 회원가입(직원 가입) 신청 후 아직 최고 관리자의 승인 절차가 완료되지 않았을 때 표시됩니다. |
| `ACCOUNT_REJECTED` | "계정 승인이 거절되었습니다. 관리자에게 문의하세요." | 신청된 계정 가입이 최종 반려(거절)된 이력을 가질 때 로그인 불가 사유로 표시됩니다. |


### 5-2. 직원 계정 초대 시스템 관련 에러
이 에러는 주로 초대 URL을 타고 접속했을 때나, 비밀번호를 초기 세팅해야 하는 계정 인증 단계에서 표시됩니다.
| 에러 코드 (`AuthErrorCode`) | 실제 출력 텍스트 | 발생 조건 및 상세 흐름 |
| --- | --- | --- |
| `ACCOUNT_INVITED` | "비밀번호를 먼저 설정해주세요. 이메일을 확인해주세요." | 직원으로 초대를 받은 상태이지만, 링크를 통해 비밀번호를 1회 셋업하지 않아 임시 락(Lock)이 걸린 상태입니다. |
| `INVITATION_NOT_FOUND` | "유효하지 않은 초대 링크입니다." | 발송된 가입/비밀번호 생성 링크 URI가 깨졌거나 데이터베이스상 존재하지 않는 토큰일 때 표시됩니다. |
| `INVITATION_EXPIRED` | "초대 링크가 만료되었습니다." | 이메일로 발송된 직원 초대 링크의 유효시간이 지나 클릭했을 때 발생합니다. |


### 5-3. 2차 인증(MFA) 및 인프라·세션 한계 에러
보안 토큰 및 전산망 인증 관련 방어 로직입니다.
| 에러 코드 (`AuthErrorCode`) | 실제 출력 텍스트 | 발생 조건 및 상세 흐름 |
| --- | --- | --- |
| `MFA_REQUIRED` | "2차 인증이 필요합니다." | 에러가 아니며, 1차 비밀번호 검증이 통과되어 **추가 이메일 OTP 창 컴포넌트로 전환하라는 내부 렌더링용 플래그(Sign)** 역할입니다. |
| `MFA_INVALID` | "인증 코드가 올바르지 않습니다." | 이메일로 수신된 6자리 OTP를 다르게 기입하고 검증(Submit)을 시도했을 때 발생합니다. |
| `MFA_EXPIRED` | "인증 코드가 만료되었습니다." | 입력 가능 제한 시간(5분)을 초과한 후, 만료된 기존 OTP 핀번호를 들고 검증을 시도할 때 반환됩니다. |
| `SESSION_EXPIRED` | "세션이 만료되었습니다. 다시 로그인해주세요." | 장시간 마우스 이벤트나 서버 통신 없이 방치되어 암호화 토큰 기간(30분)이 지나 강제로 로그인 화면으로 튕길 때 노출됩니다. |
| `TOO_MANY_ATTEMPTS` | "로그인 시도 횟수를 초과했습니다." | 다발적인 부정 인증을 시스템이 감지하여 네트워크 단에서 일시적으로 응답을 블록(Block)할 때 방어막으로 출력됩니다. |
| `NETWORK_ERROR` | "네트워크 오류가 발생했습니다." | 서버 오프라인, 네트워크 환경 단절 등 API 서버로부터 응답 값을 정상적으로 페치(Fetch) 받지 못하는 전산 장애 시 활용됩니다. |

## 6. 관리자 테스트 계정 목록표

다음은 어플리케이션 구동 시 사용해볼 수 있는 초기 테스트 계정입니다.

| 사용자 식별명 / 이메일 | 비밀번호 | 권한 구분 | 2차 인증(MFA) 방식 |
| --- | --- | --- | --- |
| `colin@cntt.co.kr` | Admin123! | admin(관리자) | 활성화됨 / 이메일 인증 필수 |
| `manager@cnttech.co.kr` | Manager123! | manager(매니저) | 비활성 (바로 로긴) |
| `viewer@cnttech.co.kr` | Viewer123! | viewer(조회자) | 비활성 (바로 로긴) |
| `locked@cnttech.co.kr` | Locked123! | viewer | 잠김 상태 테스트 |


> **안내사항**: 
> 2차 인증 계정(예: `colin@cntt.co.kr`)으로 로그인할 시 실제 이메일 전송 대신, 브라우저의 개발자 도구의 콘솔(Console) 창에 Mock OTP 코드가 즉시 출력(`[이메일 발송 시뮬레이션]`)됩니다. 해당 6자리 숫자를 복사해 입력 창에 타이핑하면 기능 테스트를 수행하실 수 있습니다.

## 요약

이 `LoginPage.tsx`는 **"접근 제어 강화"**와 **"사용자 경험(UX) 피드백"**을 주요 목적으로 디자인되어 있습니다.
Zustand를 통한 인증 상태 모니터링(`isAuthenticated`, `isMfaRequired`)에 강하게 결합되어 있으며, 개발자는 필요시 이 컴포넌트를 수정하지 않고 `authService.ts`와 `authStore.ts`의 로직만 분리 변경하여 실 서버 API 연동 확장을 수행할 수 있어야 합니다.
